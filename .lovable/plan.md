# Fix: `cn_payment_pkey` Constraint Violation on Payment Save

## Root Cause

`cn_payment.payment_sequence_no` is defined as `bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` — a **global auto-incrementing** primary key across all payments.

The `public_api_payment_save` RPC explicitly inserts `v_sort_idx` (1, 2, 3...) into `payment_sequence_no` (line 82 of the current migration). Because it uses `GENERATED BY DEFAULT`, the explicit value **overrides** the identity sequence. This means every call tries to insert `payment_sequence_no = 1, 2, 3...` which collides with existing rows.

## Fix

**One migration, one file changed:**

### 1. Database Migration — Update `public_api_payment_save` RPC

Remove the explicit `payment_sequence_no` column from the `INSERT INTO cn_payment` statement. Let the IDENTITY column auto-generate the next value.

**Before (line 80-100):**

```sql
INSERT INTO cn_payment (
  payment_id,
  payment_sequence_no,  -- ← THIS CAUSES THE COLLISION
  payment_amount, payment_code, mop_code, period, fund_code,
  base_currency, currency_conversion_rate
) VALUES (
  v_payment_id,
  v_sort_idx,           -- ← Hardcoded 1, 2, 3...
  ...
);
```

**After:**

```sql
INSERT INTO cn_payment (
  payment_id,
  -- payment_sequence_no omitted — auto-generated by IDENTITY
  payment_amount, payment_code, mop_code, period, fund_code,
  base_currency, currency_conversion_rate
) VALUES (
  v_payment_id,
  ...
) RETURNING payment_sequence_no INTO v_seq_no;
```

The `RETURNING` clause captures the generated value so it remains available if needed downstream.

Additionally, resync the identity sequence to avoid collisions from prior manual inserts:

```sql
SELECT setval(
  pg_get_serial_sequence('cn_payment', 'payment_sequence_no'),
  COALESCE((SELECT MAX(payment_sequence_no) FROM cn_payment), 0)
);
```

### Summary


| Item                | Detail                                                              |
| ------------------- | ------------------------------------------------------------------- |
| **Root cause**      | RPC inserts explicit `v_sort_idx` (1,2,3) into a global IDENTITY PK |
| **Fix**             | Omit `payment_sequence_no` from INSERT; let IDENTITY auto-generate  |
| **Sequence resync** | Reset identity sequence to current MAX to avoid gaps/collisions     |
| **Files**           | 1 new migration (drops + recreates `public_api_payment_save`)       |
| **Risk**            | None — no schema change, no UI change, backward-compatible          |
