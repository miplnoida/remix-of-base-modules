

# Root Cause: `cn_payment` INSERT reads `payment_code` from wrong array

## The Bug

The `cn_payment` INSERT (line 221-249 in the current RPC) iterates over `p_methods` and reads:
```sql
v_meth->>'payment_code'   -- NULL! Methods don't have payment_code
v_meth->>'fund_code'      -- NULL! Methods don't have fund_code
```

**Methods** represent payment modes (Cash, Cheque, Card) and contain: `mop_code`, `currency_code`, `original_amount`, `bank_code`, etc.

**Components** represent contribution line items and contain: `payment_code`, `fund_code`, `amount`, `period`.

The working version (migration `20260326160343`) used a **cross-product allocation loop** that correctly pulled `payment_code` / `fund_code` from `v_comp_arr[v_comp_idx]` while pulling `mop_code` from `v_meth_arr[v_meth_idx]`, allocating amounts across both arrays.

The `20260410` migration broke this by replacing the cross-product with a simple `FOR v_meth IN p_methods` loop, losing the component linkage. The `20260412` fix inherited this same structural bug.

## Why it wasn't caught before

The `20260410` version also had the `period::timestamp` type-cast bug, which caused ALL payments to fail before reaching the `payment_code` NOT NULL constraint. The `20260412` fix corrected the cast but exposed this pre-existing structural defect.

## Fix Plan

**Single migration** — recreate `create_c3_payment_with_receipt` restoring the cross-product allocation logic from `20260326160343`, while keeping the two fixes from `20260412`:

1. **Restore cross-product allocation**: `cn_payment` rows are generated by iterating both `p_components` and `p_methods` arrays simultaneously, reading `payment_code`/`fund_code`/`period` from components and `mop_code`/`bank_code`/etc from methods, with proper amount splitting when multiple methods pay multiple components.

2. **Keep timestamp cast**: Period values cast with `::timestamp`.

3. **Keep sequence_no-aware overpayment guard**: Guard joins `c3_payment_components` filtered by `sequence_no`.

4. **Keep all other fixes**: `is_for_director` flag, advisory lock, `c3_payment_components` insert loop, receipt creation — all unchanged.

### Technical detail

The cross-product loop allocates like this:
```text
Components: [SSE $70, LVC $30]
Methods:    [Cash $100]

Result cn_payment rows:
  Row 1: payment_code=SSE, mop_code=CSH, amount=$70
  Row 2: payment_code=LVC, mop_code=CSH, amount=$30
```

This ensures every `cn_payment` row has a valid `payment_code` from the component and a valid `mop_code` from the method.
