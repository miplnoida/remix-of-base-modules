

# Fix: Receipt Number Bug + Payment History Visibility

## Two Root Causes Found

### Bug 1: Receipt Number Always `RCP-2026-000001`

The `public_api_payment_save` RPC manually generates a receipt number using a broken counting query (`WHERE receipt_number LIKE 'RCP-...'`), but the system already has an **AFTER INSERT trigger** (`trg_set_receipt_number`) on `cn_receipt` that generates the correct receipt number using the configured `payment_module_config` segments.

What happens today:
1. RPC inserts `receipt_number = 'RCP-2026-000001'` (broken count â€” always 1)
2. Trigger fires AFTER INSERT, overwrites `receipt_number` with the correct configured format
3. **But** the RPC returns `v_receipt_number` which still holds the old broken value
4. C3-Wizard stores the wrong receipt number; database has the correct one

**Fix**: Remove manual `receipt_number` generation from the RPC entirely. Insert `receipt_number` as `NULL` (trigger will populate it). After insert, SELECT the trigger-generated `receipt_number` back and return it in the response.

### Bug 2: API-Posted Payments Not Visible on Payment History

The Payment History page (`/cashier/payment-history-mgmt`) filters:
```
.or('status.is.null,status.eq.active')
```

But `public_api_payment_save` inserts headers with `status = 'P'` (Processed). These records are filtered out â€” invisible on screen.

**Fix**: Add `status.eq.P` to the filter so API-posted payments appear alongside internally created ones.

## Implementation

### Step 1: Migration â€” Fix `public_api_payment_save` RPC

- Remove `v_receipt_number` generation logic (the `'RCP-' || ...` block)
- Insert `cn_receipt` with `receipt_number` as `NULL` â€” the existing `trg_set_receipt_number` trigger will set it
- After the INSERT...RETURNING, SELECT the updated `receipt_number` from `cn_receipt` (since the trigger fires AFTER INSERT and does an UPDATE)
- Return the trigger-generated receipt number in the response

### Step 2: Fix Payment History Filter

In `src/pages/cashier/PaymentHistoryManagement.tsx` line 199, change:
```typescript
// FROM:
.or('status.is.null,status.eq.active')
// TO:
.or('status.is.null,status.eq.active,status.eq.P')
```

This ensures payments posted via the public API (status `'P'`) appear in the history grid.

### Files Modified

1. **New SQL migration** â€” Recreates `public_api_payment_save` without manual receipt number generation; relies on the existing trigger
2. **`src/pages/cashier/PaymentHistoryManagement.tsx`** â€” Adds `status.eq.P` to the payment history query filter
