
Goal: fix the C3-Wizard payment repost failure without changing the existing business meaning of payment sequencing for normal cashier/internal payment flows.

What I found:
- The current public API RPC `public_api_payment_save` explicitly inserts `payment_sequence_no = v_seq` starting from `1`, which is the direct cause of the duplicate PK error.
- The `cn_payment.payment_sequence_no` column is already `bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`, and internal cashier flow `create_payment_with_receipt` intentionally does not pass `payment_sequence_no`; it lets the database auto-generate it.
- Existing code across cashier/history/verification treats `payment_sequence_no` as a globally unique row identifier for `cn_payment`, not as a per-payer/per-period partial-payment counter.
- Therefore, reusing `payment_sequence_no` as a business sequence for partial payments would risk breaking existing logic.

Implementation plan:
1. Update `public_api_payment_save` to stop inserting `payment_sequence_no`
   - Remove `payment_sequence_no` from the `INSERT INTO cn_payment (...)` column list and values.
   - Let the identity sequence generate a unique value exactly like the internal `create_payment_with_receipt` flow.
   - Keep the loop index only for counting inserted rows for `cn_receipt.total_number_of_payments`.

2. Preserve all existing external payment behavior
   - Keep the numeric `payment_id` fix already introduced.
   - Keep receipt generation unchanged (`receiptId`, `receiptNumber`, `receiptTotal` response contract).
   - Keep current payload structure and edge function unchanged.
   - Keep current `period` handling unless separately approved for correction.

3. Add migration safety for the identity sequence
   - Include `SELECT setval('cn_payment_payment_sequence_no_seq', (SELECT COALESCE(MAX(payment_sequence_no), 0) FROM public.cn_payment));`
   - This ensures the underlying sequence is aligned before new inserts, avoiding collisions if prior manual inserts affected sequence state.

4. Do not implement payer/period/schedule-based partial-payment numbering in `payment_sequence_no`
   - Based on the current schema and usage, `payment_sequence_no` is technical row identity, not the employer partial-payment sequence.
   - If partial-payment business sequencing is needed later, it should be introduced in a separate dedicated field/table tied to payer + period + schedule, not by changing this primary key behavior.

5. Validation after the fix
   - Confirm reposted payments can insert successfully through `/api/v1/api/payment/save/{payerId}/{payerType}`.
   - Confirm `cn_payment` rows now receive unique auto-generated `payment_sequence_no` values.
   - Confirm receipt creation still succeeds and response payload still returns canonical IDs.
   - Confirm no internal cashier flow is affected because it already relies on identity generation.

Expected file change:
- New SQL migration recreating `public_api_payment_save` safely and resetting `cn_payment_payment_sequence_no_seq`.

Technical notes:
- Safe pattern to follow is the same one already used in `create_payment_with_receipt`:
```text
cn_payment_header.payment_id  -> manually generated bigint with advisory lock
cn_payment.payment_sequence_no -> database identity
cn_receipt.receipt_id -> database identity
```
- This resolves the C3-Wizard error with minimal blast radius and avoids altering established payment/verification/reporting logic that depends on globally unique `payment_sequence_no`.

Communication back to C3-Wizard team after deployment:
- The issue was caused by the API explicitly forcing `payment_sequence_no = 1`.
- The API has been corrected to use the database-generated payment detail sequence.
- Repost can proceed.
- `payment_sequence_no` should be treated as an internal line identifier, while `paymentId`, `receiptId`, and `receiptNumber` remain the important returned references.
