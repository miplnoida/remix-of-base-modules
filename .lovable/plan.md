# Fix: Return `receipt_number` in Payment Save Response + Confirm `paymentReferenceId` Storage

## Analysis

### Issue 1 — `receipt_number` Not Returned

**Root cause**: The `public_api_payment_save` RPC (line 661-669) returns `payment_id`, `receipt_id`, `total`, `line_count`, `payment_reference_id`, and `is_for_director` — but **omits `receipt_number**`.

The `receipt_number` IS correctly generated by the `trg_set_receipt_number` trigger on `cn_receipt` INSERT. Verified in DB: payment_id=222 has `receipt_number = 'REC-138-140420260756'`.

**Fix**: After the `cn_receipt` INSERT, SELECT the generated `receipt_number` back and include it in the return JSON.

### Issue 2 — `paymentReferenceId` Storage

**Already handled**: Line 596 reads `paymentReferenceId` from the payload, and line 608 stores it in `cn_payment_header.payment_reference_id`. No change needed.

## Implementation

### Step 1: Update `public_api_payment_save` RPC

**Migration SQL** — single change to the RPC:

1. After the `cn_receipt` INSERT (line 657-659), add a SELECT to fetch the trigger-generated `receipt_number`:

```sql
SELECT receipt_number INTO v_receipt_number
FROM cn_receipt WHERE receipt_id = v_receipt_id;
```

2. Add `receipt_number` to the return object (line 661-669):

```sql
RETURN jsonb_build_object(
  'success', true,
  'payment_id', v_payment_id,
  'receipt_id', v_receipt_id,
  'receipt_number', v_receipt_number,
  'total', v_total,
  'line_count', v_line_count,
  'payment_reference_id', v_payment_reference_id,
  'is_for_director', v_is_for_director
);
```

The variable `v_receipt_number` is already declared (line 553).

### Step 2: No Edge Function Changes

The `handlePaymentSave` function (line 787-792) returns `data` directly from the RPC, so the new field flows through automatically.
