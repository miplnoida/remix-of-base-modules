# Critical Bug Fix: Payment Save API — Type Mismatch (payment_id bigint vs text)

## Problem Confirmed

The `public_api_payment_save` RPC generates text-based payment IDs (`PMT-2026-000001`) but the `cn_payment_header.payment_id` column is `bigint GENERATED BY DEFAULT AS IDENTITY`. PostgreSQL cannot cast text to bigint, causing every C3-Wizard payment call to fail with error code 42804.

**Impact**: All online payments from C3-Wizard are lost at the database level (charged via CyberSource but never recorded).

## Root Cause Analysis


| Component                                    | Expected Type                        | Actual Value             |
| -------------------------------------------- | ------------------------------------ | ------------------------ |
| `cn_payment_header.payment_id`               | `bigint`                             | `PMT-2026-000001` (text) |
| `cn_payment.payment_id`                      | `bigint NOT NULL`                    | Same text value          |
| `cn_receipt.payment_id`                      | `bigint NOT NULL`                    | Same text value          |
| Internal RPC (`create_payment_with_receipt`) | Uses `INTEGER` variable, numeric IDs | Works correctly          |


The internal payment system (`create_payment_with_receipt`) correctly uses `MAX(payment_id) + 1` to generate numeric IDs. The public API RPC introduced a conflicting text-based format.

## Recommended Fix: Option 2 — Generate Numeric Payment IDs

**Why not Option 1 (change column to text)?** The `payment_id` column is referenced as `bigint` across `cn_payment`, `cn_receipt`, `c3_payment_components`, `cheque_verification`, batch RPCs, and the internal `create_payment_with_receipt` RPC. Changing the type would require altering every related table, FK constraint, RPC variable declaration, and join — a high-risk change.

**Fix**: Modify `public_api_payment_save` to use the same numeric ID pattern as the internal RPC, using `MAX(payment_id) + 1` with an advisory lock for concurrency safety.

## Implementation Steps

### Step 1: Create Migration — Replace `public_api_payment_save` RPC

Replace the text-based ID generation:

```sql
-- OLD (broken)
v_payment_id TEXT;
v_payment_id := 'PMT-' || TO_CHAR(NOW(), 'YYYY') || '-' || LPAD(nextval('cn_payment_seq')::TEXT, 6, '0');
```

With numeric ID generation matching the internal pattern:

```sql
-- NEW (fixed)
v_payment_id BIGINT;
PERFORM pg_advisory_xact_lock(7839202);  -- different lock key from internal RPC
SELECT COALESCE(MAX(payment_id), 0) + 1 INTO v_payment_id FROM cn_payment_header;
```

Also update `v_receipt_number` to remain text-based (receipts use text), but `payment_id` references across the INSERT statements for `cn_payment_header`, `cn_payment`, and `cn_receipt` will now correctly be bigint.

Remove the EXCEPTION fallback block that also generates text-based IDs.

### Step 2: Verify Related Tables Accept the Fix

- `cn_payment.payment_id` — `bigint NOT NULL` — compatible
- `cn_receipt.payment_id` — `bigint NOT NULL` — compatible
- No code changes needed in the edge function (`public-api/index.ts`) since it passes through to the RPC

### Step 3: Test with C3-Wizard Sample Payload

Invoke the public-api edge function with a test payment for payer 658852 to confirm the 500 error is resolved and a receipt is returned.

## Technical Details

### Files Modified

1. **New SQL migration** — Drops and recreates `public_api_payment_save` with:
  - `v_payment_id BIGINT` instead of `TEXT`
  - Advisory lock (`pg_advisory_xact_lock(7839202)`) for concurrency
  - `MAX(payment_id) + 1` for sequential numeric IDs
  - Removal of the text-based fallback in the EXCEPTION block
  - Receipt number generation remains text-based (`RCP-YYYY-NNNNNN`)
2. **No edge function changes** — The `public-api/index.ts` handler passes parameters to the RPC unchanged
3. **No table schema changes** — Column types remain as-is
